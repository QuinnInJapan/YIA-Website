Manual Generation System — 3 Approaches

Pick the architecture that fits best. All produce a Japanese Word document with TOC + automated screenshots.

A

YAML + Node.js Script (Recommended)

Stack: YAML data files → Node.js generation script (docx library) + Playwright for screenshots

How it works: One YAML file defines all manual sections with structured fields (title, intro, steps, screenshot IDs). A single npm run generate-manual command runs Playwright to capture screenshots, then builds the .docx from YAML + images.

Pros

  • Stays in the Node.js ecosystem (no Python dependency)
  • Single command generates everything
  • docx npm package gives fine-grained Word formatting control
  • Playwright already available via Next.js ecosystem

Cons

  • docx npm library is more verbose than python-docx
  • Japanese font handling needs explicit configuration
B

YAML + Python Script

Stack: YAML data files → Python generation script (python-docx) + Playwright (Node) for screenshots

How it works: Same YAML data structure, but uses python-docx for Word generation (more mature library for complex documents). Screenshots still captured via Node.js Playwright. Two-step process: capture screenshots, then generate doc.

Pros

  • python-docx is the gold standard for Word generation
  • Better Japanese typography support out of the box
  • Cleaner API for complex formatting (tables, headers, TOC)

Cons

  • Adds Python as a dependency to a Node.js project
  • Two separate tools to maintain (Node for screenshots, Python for doc)
  • Need to manage a Python virtualenv
C

YAML + Markdown → Pandoc

Stack: YAML data → generate Markdown with image refs → Pandoc converts to .docx with a reference template

How it works: A Node script reads YAML and produces a single Markdown file with screenshot image references. Pandoc converts to Word using a custom reference.docx template for Japanese styling. Screenshots via Playwright.

Pros

  • Markdown intermediate is human-readable and debuggable
  • Pandoc handles TOC generation natively
  • Reference template approach = change styles without changing code

Cons

  • Pandoc is an external system dependency (Homebrew install)
  • Less control over fine-grained Word formatting
  • Pandoc's Japanese support can be quirky with some templates
  • Harder to do custom layouts (callout boxes, annotated screenshots)
